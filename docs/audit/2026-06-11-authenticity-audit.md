## 总评(不粉饰)

项目**核心 AI 生成链路基本诚实**:真调 LLM API(非 mock)、JS/Python 有真实执行校验、失败兜底明确标注 `algorithm:'fallback'` 不伪装。这一点必须公允承认。

## ① 模拟数据 —— 部分存在(🟡 低)

### 查证为"不成立"的部分(必须澄清,避免冤枉)

- **AI 不是 mock**。`src/ai/client.ts:342` 真实 `fetch('/api/chat')` 走代理,`:395` 直连 DeepSeek `Bearer` 鉴权。无任何硬编码假响应。
- **内置核心算法不是假数据**。`src/presets/generators.ts` 注册表里是 `bubble_sort/quick_sort/dijkstra/kmp_automaton/tarjan_scc` 等**真实算法实现**,逐操作驱动事件。

### 确实存在的部分

**预设默认输入静默兜底**:`src/presets/leetcode.ts:4` `DEFAULT_NUMS = [2,7,11,15]`。当用户输入解析失败时,`parseInput` 不报错,直接用这组默认数据跑(`:14`、`:29`、`:32` 三条兜底路径)。用户若输入了非法数据,看到的是 `[2,7,11,15]` 的动画,**可能误以为是自己的输入**。

- **严重性低**:仅影响 `leetcode_hot100` / `acm_templates` 两个**示例目录条目**(上一轮已诚实改名为"示例")。不影响核心 AI 路径——自定义 `two_sum` 代码不在内置注册表,会走 AI 生成器而非这个预设。
- **方案**:`parseInput` 兜底到 DEFAULT 时,在动画首步加一条 `scene.note`:"输入无法解析,已用示例数据演示" —— 复用上一轮 P6 截断提示的同款模式。

## ② 简化实现 —— 存在(🟠 中)

### 2.1 统计面板的数字是估算,不是真实计数【主要问题】

`比较/交换/访问` 三个数字的来源分两种,**都不是算法真实执行计数**:

- **AI 生成器**:`src/sandbox/builder.ts:25` `statDelta()` 按**可视化事件类型**推导——`compare` 事件记 +1 比较 +2 访问,`swap` 记 +1 交换。逻辑自洽(可视化了几次就算几次),但等于"按 AI 发了多少事件计数",不是算法真实复杂度。
- **leetcode 示例**:`src/presets/leetcode.ts:87` 直接硬编码 `stats: { comparisons: i + 1, swaps: 0, accesses: i + 1 }`。two_sum 用**哈希表**,根本不存在"逐个比较 i+1 次"——这个数字是**编造的估算**,语义错误。

`src/pages/Visualizer/InfoPanel.tsx:231-233` 把这些数字以 `比较 / 交换 / 访问` 标签直接展示,**无任何"估算"标注**,呈现为精确计数。

> 公允说明:内置**排序/图算法**(`generators.ts`)的 stats 是 `++comps` / `acc += 2` **真实累加**(`:25`、`:44`),是准的。问题仅限 AI builder 的事件推导 + leetcode/acm 示例的硬编码。

- **严重性中**:数字看似精确实为近似,对"算法复杂度直观感受"有误导,但不影响动画正确性。
- **方案**:① 把 leetcode/acm 示例的硬编码 stats 改为真实累加(成本低,就在生成循环里数);② AI builder 的 stats 面板加副标题"基于可视化操作估算"或改标签为"可视化操作数",停止伪装精确。

### 2.2 recovered globals:把未定义变量当 0【掩盖性简化】

`src/sandbox/executeGenerator.ts:33-44`:AI 生成器引用未声明变量时,系统不报错,而是注入 `let ${name} = 0`(`:55`)重试,最多 4 个。

- **风险**:AI 漏写一个累加器声明(本是 bug),系统塞 0 让它**跑完并产出"成功"动画**,但语义可能已错。
- **缓解**:有完整测试覆盖(`executeGeneratorCoverage.test.ts` 5 用例,含上限/重复/TypeError 边界),是**有意的工程权衡**而非疏忽;且下游 `verifyAndTag` 在 JS/Python 可校出结果错误。
- **残留风险**:当 verify 走 @expect 或 skip 时(见 ④),语义错误的动画会带着"成功"状态漏过。
- **严重性中**:权衡合理,但缺一条"本次注入了 N 个未定义变量"的可见提示——用户无从知道动画建立在被修补的代码上。
- **方案**:`recoveredGlobals.size > 0` 时在 `GeneratorResult` 带回 `recoveredCount`,UI 降一档可信度提示("AI 代码有 N 处未声明变量已按 0 处理")。

## ③ 偶尔执行 —— 存在(🟠 中)

### 查证为"不成立"的部分

- **live-regen 无竞态 bug**。`src/hooks/useAIGenerator.ts:440-464` 有 `cancelled` 标志 + `clearTimeout` 清理,输入快速变化时旧请求被正确丢弃。✓

### 确实存在的部分

**Python 真值校验是"尽力而为"的偶发执行**:`verifyAndTag`(`useAIGenerator.ts:54`)对 Python 调 `runUserPySandboxed`,依赖 `src/sandbox/pythonWorker.ts:4` 从 jsdelivr CDN 加载 Pyodide(~6-10MB)。

- 首次加载慢(计划设 25s 超时)、CDN 不可达、断网 → `truth.ok === false` → **静默落回 @expect**(`:60`)。
- 后果:**同一段 Python 代码,网络好时走真值校验(强),网络差时走 AI 自证(弱),用户看到的徽标却可能都是"已通过校验"**。校验强度随环境漂移,但 UI 不暴露这一点。
- **严重性中**:降级本身是正确设计(不该阻塞动画),问题在于**降级不透明**——徽标 source 字段区分了 `js-exec/py-exec/expect`,但若 py 降级到 expect,显示的是 `passExpect`,用户不知道"本想真实执行但没跑成"。
- **方案**:py 真值"尝试过但失败降级"时,徽标加一档中性提示("真实执行不可用,已用 AI 推演校验"),与"本就只能 AI 推演"区分开。

## ④ 兜底欺骗 —— 存在(🔴 中高)

### 查证为"不成立"的部分(澄清)

- **失败兜底场景不欺骗**。`src/ai/fallbackScene.ts:18` `buildFallbackScene` 明确标 `algorithm:'fallback'`(`:39`),描述写"模型暂不可用/生成结果无法解析…可调整后重试",action 用 `danger` 色。UI 如实显示失败,**没有把失败伪装成成功**。这是教科书式的诚实降级,应肯定。

### 确实存在的部分【本报告最需警惕项】

**@expect 自证 + 绿色"已校验"徽标,给用户超出实际的信心。**

校验强度三档(`verifyAndTag`):
1. JS 代码 → 浏览器真实执行原函数比对(强,可信)
2. Python 代码 → Pyodide 真实执行(强,但常降级,见 ③)
3. **C++ / Java / 其他语言 → 永远只能走 `@expect`**(`verify.ts` `verifyAgainstExpect`)

第 3 档的致命点:`@expect` 是 **AI 自己声明的期望值**,生成器也是 **AI 自己写的**。若 AI 对算法理解错了,它声明的 @expect 和它写的生成器结果**会一致地错**,校验"通过",`VerificationNotice` 显示绿色 `passExpect`:**"已通过一致性校验(AI 推演结果比对)"**。

- 用户输入 C++ 代码(竞赛选手高频场景),拿到绿色"已通过校验"——但这只是 **AI 和自己对答案**,毫无独立性。
- 文案虽含"AI 推演"字样,但绿底 ✓ + "已通过一致性校验"的措辞,对非专业用户传递的是"这个动画被验证过、可信",**实际可靠性远低于绿色徽标的暗示**。
- 叠加 ② 的 recovered globals:语义错误的 C++ 动画完全可能带着绿色徽标展示。

- **严重性中高**:不是数据造假,而是**校验可信度的虚标**——最隐蔽,因为它发生在"质量保证"功能本身。命中真实高频场景(C++/Java)。
- **方案**:
  1. **@expect 路径不给绿色 ✓**。改为中性/警告色徽标:"⚠ 结果未经独立验证(基于 AI 自报期望)",与 js-exec/py-exec 的绿色"真实执行已验证"在**颜色和措辞上明确分级**。
  2. 文案去掉"已通过一致性校验",@expect 档改为"AI 自查一致"——诚实表达"这是 AI 自证,非独立验证"。
  3. 长期可选:对 C++/Java 引入轻量真值路径(如 WASM 版编译器或服务端执行),把第 3 档从"自证"升级为"真值"。

## 一句话结论

**项目没有数据造假级别的欺骗,核心 AI 链路诚实**;但存在一类系统性问题——**"质量/精确度信号"的虚标**:统计数字伪装精确、@expect 自证伪装成独立校验、Python 降级不透明。这些不会让动画"画错",但会让用户**高估动画的可信度与精确度**。修复成本普遍很低,P0 的徽标分级尤其应优先做——它正发生在用户最依赖的"这个动画可信吗"的判断点上。
